Snowflake University: senior-level deep dive

Companion to the visual guide and runbook. Each section explains why Snowflake built the feature, what problem it solves, backend behavior at a high level, cost levers, repo paths, and how to run and observe.

How to use this doc with the repo

  • Run scripts in order from the runbook (step 17 = external tables).
  • Keep Query Profile open for performance work; use History for bytes scanned and time.
  • dbt: from dbt/ with DBT_PROFILES_DIR=. and ../.venv-snowflake/bin/dbt.

DMS / Snowpipe / dbt spine

Why: Enterprise pattern: replication (DMS), durable landing (S3), governed ingest (Snowpipe), modeling (dbt), orchestration (Airflow).

Repo: README.md, infra/snowflake/, scripts/verify_snowpipe.sql, dbt gold models.

Run: cd dbt && DBT_PROFILES_DIR=. ../.venv-snowflake/bin/dbt run --select gold

Retail realtime stack

Why: Event-driven ELT: UI → Kafka → Spark Streaming → S3 → Airflow bronze.

Run: ./scripts/start_retail_stack.sh — UI :8502, Kafka :8081, Airflow :8080, DAG retail_raw_to_bronze.

1. Foundation dataset

Why: Features only show real tradeoffs at meaningful scale.

Backend: Large inserts write micro-partitions; metadata tracks min/max for pruning.

Cost: Edit row targets in script; smaller WH for dev; drop unused clones.

Repo: scripts/00_create_foundation_dataset.sql

SELECT COUNT(*) FROM RAW_DB.PUBLIC.RAW_ORDERS;

2. Transient tables

Why: Cheap staging without full durability/clone semantics of permanent tables.

Repo: scripts/01_transient_tables.sql, DAG transient_table_pipeline

Observe: INFORMATION_SCHEMA.TABLE_STORAGE_METRICS, SHOW TABLES in STAGING_DB / SILVER_DB.

3. Streams (CDC)

Why: Durable change cursor tied to table versions, not ad-hoc diffs.

Backend: Stream stores an offset; consumption advances after successful DML.

Cost: Steady consume schedule; single consumer; selective merges.

Repo: scripts/02_streams.sql, 02_streams_trigger_changes.sql, 02_streams_consume.sql, DAG stream_cdc_pipeline

4. Tasks

Why: SQL-native scheduling inside the warehouse.

Repo: scripts/03_tasks.sql, DAG task_dag_pipeline

Observe: SHOW TASKS IN SCHEMA AUDIT_DB.PUBLIC, PIPELINE_RUNS, TASK_HISTORY.

5. Dynamic tables

Why: “Always fresh enough” without hand schedules for every layer.

Cost: Deterministic SQL; tune TARGET_LAG; watch full vs incremental refresh.

Repo: scripts/04_dynamic_tables.sql

6. Materialized views

Why: Precomputed hot paths without re-scanning huge bases.

Cost: Create only for high reuse; monitor maintenance vs savings.

Repo: scripts/05_materialized_views.sql

7. Snowpark

Why: Push Python close to data; governance and performance vs local pandas.

Repo: snowpark/snowpark_pipelines.py, snowpark/udfs/, snowpark/stored_procs/

PYTHONPATH=$(pwd) .venv-snowflake/bin/python snowpark/snowpark_pipelines.py

8. dbt snapshots (SCD2)

Why: History when sources mutate.

Repo: dbt/snapshots/*, scripts/08_trigger_snapshot_changes.sql

9. Clustering benchmark

Why: Pruning is the main lever for scanned bytes.

Repo: scripts/09_clustering_benchmark.sql, BENCHMARKS.PUBLIC

10. Internals: clones, cache, monitors

Why: Fast dev copies, honest benchmarking, credit guardrails.

Repo: scripts/10_internals_deep_dive.sql

11. Iceberg on S3

Why: Open format files + metadata on customer lake; Snowflake as control plane.

Repo: scripts/11_iceberg_s3_lab.sql, snowflake_iceberg_lab.md

12. External tables and stages

Why: SQL over curated lake files without full COPY.

Backend: Stage = URL + integration; external table = file metadata; queries decode at runtime; REFRESH / AUTO_REFRESH.

Cost: Align partitions with WHERE; avoid SELECT *; compare to Snowpipe+native for hot paths.

Repo: scripts/12_external_tables_and_lake_patterns.sql (uses S3_DMS_INT, prefix dms/sales/orders/).

LIST @EXT_DEMO.PUBLIC.DMS_ORDERS_EXT LIMIT 20;
SHOW EXTERNAL TABLES IN SCHEMA EXT_DEMO.PUBLIC;
SELECT COUNT(*) FROM EXT_DEMO.PUBLIC.EXT_DMS_ORDERS;
