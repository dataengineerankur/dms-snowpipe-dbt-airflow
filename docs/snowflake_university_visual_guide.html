Pipeline 1

Postgres to Snowflake analytics spine

Source tables land in S3 through DMS, Snowpipe ingests into RAW, dbt reshapes into analytics models, and Airflow orchestrates runs.
Why this pipeline pattern
Separation of replication (DMS), durable landing (S3), governed ingest (Snowpipe), semantic modeling (dbt), and orchestration (Airflow) mirrors how enterprises reduce blast radius and assign ownership.
Cost levers
Right-size warehouses, avoid tiny parquet files in S3, monitor pipe history for failures and replays, and keep RAW thin so you do not pay storage for derived logic.
Flow
Postgres -> DMS -> S3 -> Snowpipe -> RAW -> dbt -> GOLD -> Airflow
What to observe
S3 object arrival, pipe history, dbt graph, Airflow DAG runs.
What to understand
Landing raw data is different from modeling business data.
Run and verify
cd dbt
DBT_PROFILES_DIR=. ../.venv-snowflake/bin/dbt run --select gold
DBT_PROFILES_DIR=. ../.venv-snowflake/bin/dbt test --select gold

SELECT * FROM RAW_DB.PUBLIC.RAW_ORDERS LIMIT 20;
SELECT * FROM ANALYTICS.GOLD.ORDERS_DAILY_AGG LIMIT 20;
Pipeline 2

Retail realtime pipeline

A checkout in the Streamlit app becomes a Kafka event, Spark Structured Streaming lands raw JSON in S3, and Airflow promotes raw objects into bronze.
Why Kafka + Spark here
Decouples the UI from analytics consumers, absorbs traffic spikes, and lands immutable raw events before bronze normalization—classic event-driven ELT.
Cost levers
Batch bronze promotion (Airflow schedule), monitor Spark executor sizing, and compact raw JSON sprawl if object counts explode.
Flow
UI -> FastAPI -> Kafka -> Spark Streaming -> S3 raw -> Airflow bronze
Where to look
Retail UI, Kafka Console, Spark status, Airflow DAG, S3 raw and bronze prefixes.

Concept 1

Foundation dataset

The whole university matters because the data is large enough to expose query planning, scan behavior, refresh maintenance, and history patterns.
Lead read: why scale matters
Micro-partition pruning, MV maintenance, and stream volume only show real tradeoffs at millions of rows. This script generates reproducible facts so Query Profile and storage metrics are trustworthy.
Cost levers
Lower row targets in the script for dev, use a smaller warehouse for iteration, drop benchmark clones when finished.
Tables
RAW_ORDERS, RAW_EVENTS, RAW_PRODUCTS, RAW_CUSTOMERS
Scale
50.1M orders, 100M events, 200K products, 500K customers
Lesson
Features only become educational when the workload is large enough to show tradeoffs.
Run and verify
scripts/00_create_foundation_dataset.sql

SELECT 'raw_orders', COUNT(*) FROM RAW_DB.PUBLIC.RAW_ORDERS
UNION ALL
SELECT 'raw_events', COUNT(*) FROM RAW_DB.PUBLIC.RAW_EVENTS
UNION ALL
SELECT 'raw_products', COUNT(*) FROM RAW_DB.PUBLIC.RAW_PRODUCTS
UNION ALL
SELECT 'raw_customers', COUNT(*) FROM RAW_DB.PUBLIC.RAW_CUSTOMERS;

Concept 2

Transient tables

Cheap staging with sharp edges. This demo shows both the good pattern and the anti-pattern.
Backend note
Transient tables still use micro-partitions but trade some durability and clone semantics for lower storage overhead—ideal for staging, risky for systems of record.
Cost levers
Promote only validated data to permanent tables; drop transient scratch objects aggressively.
Created
Transient staging clone plus permanent promoted copy.
Observe
Storage metrics, clone behavior, and the expected failure path.
Understand
Transient is great for staging, not a blanket replacement for permanent tables.
Run and verify
scripts/01_transient_tables.sql

SHOW TABLES LIKE 'STG_ORDERS%' IN SCHEMA STAGING_DB.PUBLIC;
SHOW TABLES LIKE 'STG_ORDERS%' IN SCHEMA SILVER_DB.PUBLIC;
SELECT * FROM INFORMATION_SCHEMA.TABLE_STORAGE_METRICS
WHERE TABLE_NAME IN ('STG_ORDERS_TRANSIENT','STG_ORDERS_PERMANENT');
Concept 3

Streams CDC

Streams teach you that CDC is an offset problem, not just a MERGE problem. The repo separates setup, change generation, and deterministic consumption.
Backend note
Streams are offsets into table versions, not a message broker. Consumption advances only after successful DML, which is why single-consumer designs are safer.
Cost levers
Narrow MERGE predicates, consume on a steady schedule, and avoid multiple writers racing the same stream.
Objects
ORDERS_CDC_STREAM, ORDERS_AUDIT, ORDERS_SILVER
Observe
Backlog before consume, offset advance after consume, audit growth, and silver convergence.
Understand
Single-consumer design matters. Audit-first CDC is easier to reason about.

Concepts 4 and 11

Snowflake tasks and Airflow

The task graph shows in-warehouse orchestration. Airflow acts as the outer control plane for repeatable runs and visual learning.
When to pick Tasks vs Airflow
Tasks excel when all steps are SQL inside Snowflake. Airflow wins for cross-system DAGs (this repo uses both on purpose).
Cost levers
Suspend idle graphs, avoid scanning entire facts when only deltas exist—pair with streams or incremental patterns.
Task graph
ROOT_TASK_INGEST -> staged CDC -> silver -> deletes -> gold -> run log
Airflow lesson
Use Airflow when you want cross-system orchestration or easier replay and visualization.
Understand
Only one task should consume the stream directly.

Concepts 5 and 6

Dynamic tables and materialized views

These are convenience layers with maintenance tradeoffs. They reduce manual work, but only if your SQL and access patterns align with how Snowflake maintains them.
Dynamic tables vs MVs
Dynamic tables optimize for layered pipelines with lag contracts; MVs optimize for read-heavy aggregates and filtered access paths. Both charge maintenance and storage.
Cost levers
Watch refresh mode reasons, avoid non-deterministic SQL that forces full refresh, and drop unused MVs.
Dynamic tables
Bronze, silver, and gold managed refresh chain.
Materialized views
Aggregate, filter-heavy, recent-order, and secure MV patterns.
Understand
Managed does not mean free; refresh mode and maintenance cost still matter.
Run and verify

scripts/05_materialized_views.sql

SHOW MATERIALIZED VIEWS IN SCHEMA GOLD_DB.PUBLIC;
Concepts 7 and 8

Snowpark and dbt snapshots

Snowpark teaches pushdown and lazy execution. Snapshots teach durable history when source tables keep changing.
Why Snowpark
Keeps governance and data residency inside Snowflake while giving engineers Python ergonomics; execution still runs as Snowflake queries under the hood.
Cost levers
Always read execution plans, avoid shipping huge intermediates to the client, and size warehouses for write-heavy stages.
Snowpark outputs
SP_ORDERS_ENRICHED, SP_ORDER_VALUE_SEGMENTS, SCORED_ORDERS, SP_REGION_FEATURES
Snapshot lesson
dbt_valid_from and dbt_valid_to make change history explicit.
Understand
Snowpark is a Snowflake execution client, not local pandas. Snapshots preserve truth over time.

Concepts 9 and 10

Clustering and internals

This is where Snowflake stops feeling magical and starts feeling physical: layout, pruning, clones, cache behavior, and resource monitors.
Clones and cache
Zero-copy clones share immutable files until divergence—fast for dev/test. Result cache can mask true performance; disable when benchmarking honestly.
Cost levers
Resource monitors cap spend; clustering only where predicates justify it.
Benchmark
Five physical copies of the orders table with different clustering strategies.
Internals pack
Clone database, clone table, cache comparisons, and resource monitor setup.
Understand
Cluster for predicates, not for aesthetics. Always note cache state when measuring.

Lake pattern

External stages and external tables

Query Parquet in S3 without loading bytes into native Snowflake storage first. Snowflake keeps file metadata and partition expressions; compute decodes files at query time.
Why Snowflake built this
Enterprises already land curated data in object storage. External tables let SQL consumers access those files with IAM-governed paths, partition pruning, and optional auto-refresh—without forcing an immediate full COPY.
Backend engineering (conceptual)
Stage binds a URL + storage integration to credentials. External table registers files and partition columns (often path-derived). Queries compile into scans over matching objects; Parquet is read and projected using column expressions like $1:col. REFRESH updates metadata when AUTO_REFRESH is off or events are not wired.
Benefits vs native tables
Pros: no duplicate storage, fast time-to-query on lake files, clear separation of lake vs warehouse roles. Cons: higher per-query decode cost vs well-clustered native tables; freshness depends on refresh strategy; wide scans hurt credits.
Cost optimization
Align partitions with filters (e.g., load_dt= paths), avoid SELECT *, compare Query Profile to Snowpipe-loaded ANALYTICS.RAW.ORDERS for hot paths, and promote heavy dashboards to native or MV/dynamic tables.

Concept add-on

Iceberg on S3

Use this lab to compare native Snowflake tables with Snowflake-managed Iceberg tables whose files live on S3. If you later want true multi-engine discovery, add the right external catalog or open catalog setup on top.
Iceberg vs external table vs native
Native table: best query ergonomics inside Snowflake. External table: query files in place. Iceberg: open table format with snapshot semantics—often paired with external catalogs for multi-engine sharing beyond this starter.
Cost levers
External volume IAM trust must be correct; compact small files upstream; use deterministic load slices for reproducible labs.
What it is
A Snowflake-managed Iceberg starter on your existing S3 bucket.
What it is not
Not a full cross-engine discovery setup by itself yet.
Understand
This is about open storage patterns and lakehouse direction, not replacing every native Snowflake table.
